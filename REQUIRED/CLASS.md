# Codex 开发规范（面向 Deep Research Agent 项目）

> 本文档只定义代码开发与工程协作约束，不定义 Deep Research 项目的具体业务策略、任务设计或领域规则。业务侧约束应放在独立文档中维护。  
> 本规范的目标不是让项目"看起来架构完整"，而是约束开发过程：始终从最小可验证单元出发，小步迭代，测试先行，保留现场，持续留痕，让系统在长期演进中仍然可读、可测、可恢复、可追责。

---

## 0. 核心原则

这是一个 Agent 项目，但 Agent 开发绝不是先搭一个大而全框架，再把能力往里填。默认做法必须相反：先构建**最小 harness**，先把一个最小链路跑通，再围绕真实失败案例逐步补测试、补观测、补恢复、补抽象。任何复杂设计，都必须由当前已经出现的重复问题来证明其必要性，而不是为了未来假设提前建设。

项目开发过程中，必须始终保持以下判断顺序：

第一，**不能破坏已有通过的测试**。  
第二，**新增能力必须先以最小可测试单元落地**。  
第三，**复杂度的增长必须慢于需求和失败模式的增长**。  
第四，**所有重要工作都要留痕**，包括成功路径、失败路径、尝试过但无效的方法，以及后续待办。

本项目默认偏好简单、直接、清晰、可验证的实现。不要为了"更像框架""更像平台""更优雅的架构"而过早抽象。能用函数、模块、dataclass 和清晰边界解决的问题，不要引入沉重模式。主入口可以作为 assembly root 存在，但不能无限膨胀成总控上帝类。
注意，对于一些面向未来项目管理的约束，不要一次就加上，例如”故障点恢复，而不是从头重启“这些故障重启类的。就是我们应该是从一个agent loop的while循环开始慢慢往上搭，到合适的时候加合适的东西。

## 2. 默认开发方式：先最小链路，再逐步增强

任何新能力开发都必须从最小、透明、可测的 harness 开始。所谓最小 harness，是指能以最低复杂度验证一个明确假设的最小执行链路。例如：一次模型调用、一次工具调用、一次状态更新、一次结果校验。如果该能力还不能在最小 harness 中被明确观察、测试和复现，就不应先扩展成通用框架。

默认工作顺序如下：

先阅读相关模块、已有测试和 CHANGELOG，确认当前边界与已有约束；然后先写一个最小失败测试，明确希望系统产生什么行为；接着用最小实现让测试通过；再运行快速测试集验证没有破坏现有功能；最后更新 CHANGELOG，记录本次工作、结果、问题与新增任务。

所有增强复杂度的设计——例如新的状态层、统一恢复系统、工具注册机制、planner 抽象、事件总线、通用执行框架——都必须先由失败案例、测试需求或重复出现的工程痛点证明有必要。没有证据，不做抽象；只有一次使用场景，不做框架；只是"未来可能需要"，不作为当前设计依据。

## 3. 测试优先：tests are everything

测试不是收尾工作，也不是提交前补齐的形式要求。测试是本项目最重要的工程事实来源，是行为规范、回归保护、诊断入口和设计边界的共同载体。

### 3.1 测试先行规则

所有新模块都必须**先有对应测试文件，再实现模块本身**。先写测试，明确这个模块、类或函数应该产生什么行为，再让实现去满足测试。测试不是附属物，而是接口契约的一部分。

当发现 bug 时，不允许直接修实现。必须先增加一个能稳定复现该 bug 的测试，再修复问题。这样历史问题才能成为永久回归保护。

任何提交都不得破坏当前已通过测试。不要接受"先合并，之后再修测试"的做法。测试失败的代码不得提交，不得合并。

### 3.2 测试质量要求

测试必须高质量、可重复、可定位，而不是数量堆砌。所谓高质量，至少意味着：

- 能稳定复现目标行为，不依赖脆弱偶然条件
- 错误时能快速定位失败点，而不是只告诉你"错了"
- 输出简洁，不污染上下文窗口
- 覆盖关键失败模式，而不只是最顺路径
- 对 Agent 项目中的接口契约、流程行为、恢复行为有明确约束
- 能随系统演进提供真实回归保护，而不是很容易被"修改断言"绕过

对于 CLASS 之类存在参考数据、参数点和误差验证的场景，必须在测试 harness 上投入足够精力：生成高质量参考数据、设计清晰 verifier、监控常见失败模式、在偏差出现时定位上游首个分歧模块，而不是在下游结果层反复打补丁。

当发现结果有偏差时，必须沿流水线向上追踪，找到第一个开始偏离的模块并在该处修复。不要只在最终输出层做掩盖性修补。上游修对了，下游会自动改善。

### 3.3 测试分层要求

Agent 项目不能只靠单元测试。测试体系至少应覆盖以下层级：

- **单元测试**：纯函数、解析器、状态变换、调度规则、小型数据结构行为  
- **接口测试**：工具 schema、prompt 输出结构、序列化/反序列化、错误分类对象  
- **流程测试**：最小 harness 下的一轮或多轮 agent 执行路径  
- **回归测试**：历史 bug、异常输入、边界条件、曾失败过的真实案例  
- **评估测试**：任务级成功率、恢复率、步数、调用次数、成本或延迟指标

新增功能时，至少应考虑它属于哪一层测试边界。如果某个改动会影响 prompt、tool schema、state schema 或执行流程，单元测试通常不足，必须增加接口测试、流程测试或回归测试。

### 3.4 Prompt / Tool / State 契约测试

对 Agent 项目来说，最脆弱的地方往往不是"算法"，而是契约漂移。prompt 变了，模型输出结构变了；工具参数名变了，调用断了；状态结构变了，checkpoint 不能恢复。因此以下改动必须有契约测试：

- prompt 模板或输出格式变更
- 工具输入输出 schema 变更
- 状态结构变更
- planner / executor 之间协议变更
- checkpoint 格式或恢复逻辑变更

禁止无测试地修改这些契约层内容。禁止在没有兼容性验证的前提下任意调整字段名、结构层级、解析方式。

### 3.5 快速测试模式与上下文卫生

LLM 天然容易"时间失明"，会在不必要时长时间跑整套测试。因此每个测试文件都必须支持 `--fast` 模式，通过 pytest fixture 实现。`--fast` 必须运行一个**确定性的、约 10% 的子样本**；对于扰动等更重测试，可降到约 5 个具有代表性的 k-modes，但仍要覆盖完整范围。

`--fast` 的抽样必须是确定性的，且最好能基于 agent id、测试名或稳定 seed 变化，以保证同一 agent 内部可复现，不同 agent 之间又不会永远只测同一小片区域。

默认开发循环是：每次改动后运行快速测试；只有在提交前或必要里程碑时，再运行完整测试集。严禁在日常迭代中高频跑整套慢测试，浪费上下文和时间。

测试输出必须简洁。成功时输出最多 5–10 行，失败时控制在约 20 行。默认使用 `pytest -q`。不要把大数组、长文本、完整响应直接打印到 stdout。详细诊断应写入 `test_logs/` 或其他专门日志文件。数组比较只打印聚合统计，例如最大相对误差、误差出现的位置、整体通过率，而不是完整内容。错误信息必须可 grep，例如把 `ERROR` 和失败原因放在同一行。

你的测试不一定是针对全体的测试，可以只是针对你新增模型的测试，看看输入输出是否符合预期。例如你新增一个检索模块，你需要真实跑通这个检索并看看检索回来的东西。
### 3.6 提交前测试要求

每次提交前，必须至少运行：

```bash
pytest tests/ -q --fast
```

如果失败，必须先修复，再提交。绝不允许"先提交，后续再补修"。

如果新功能确实需要改变原有行为，必须明确更新相应测试，使测试准确表达新行为。不要直接删测试，也不要用 skip 绕开问题，除非跳过本身就是经过说明和记录的显式工程决策。

CHANGELOG 中应持续记录测试通过率、覆盖范围和关键评估指标的变化，例如哪些模块已通过、哪些尚未通过、扰动测试当前通过情况如何、恢复测试是否有回归等。

## 4. 版本管理与仓库纪律

项目使用 Git 进行版本管理，远端仓库为：

```
git@github.com:V-rand/open_niumasearch.git
```

提交必须保持小规模、可测试、语义明确。一次提交只做一件事，但这里的"一件事"允许包含同一工作单元所需的最小测试、实现、相关文档和 CHANGELOG 更新。换句话说，修一个 bug 的提交可以同时包含：复现 bug 的测试、修复实现、更新 CHANGELOG。这仍然是一件事。

不要把多个无关模块、多个方向的重构、多个实验性想法塞进一个提交。若需要跨多个文件重构，应尽量把"纯重构"和"功能变化"拆分为独立提交。功能提交应可被测试证明，重构提交应尽量不改变行为。

### 4.1 禁止上传的内容

任何包含密钥、凭据、私密配置、本地环境状态或运行过程噪音的内容，都不得上传到 GitHub。包括但不限于：

- `.env` 及其他包含密钥的配置文件
- 本地虚拟环境、uv 环境目录
- 缓存目录、临时目录、构建产物
- agent 运行过程中下载到本地的原始资料
- agent 在执行过程中生成的中间 scratch 文件
- 运行时日志、trace、checkpoint、快照（除非该类文件被明确设计为应进入版本库）
- 其他根据工程经验应视为本地过程文件的内容

默认原则是：仓库只保存源代码、测试、必要文档、必要配置模板和经过明确约定应版本化的静态资源。任何"运行时自然产生"的过程文件，默认不进仓库。

## 5. CHANGELOG.md：工作留痕是强制要求

本项目要求持续维护 CHANGELOG.md，它不是可有可无的说明文件，而是对 Codex 尤其重要的执行轨迹记录。只要完成了一个有意义的工作单元，就必须更新 CHANGELOG.md。

所谓"有意义的工作单元"，指至少满足以下之一：

- 新增或修改了一个测试文件
- 修复了一个 bug
- 新增了一个函数、类、模块或工具接口
- 调整了状态结构、恢复策略或序列化结构
- 进行了会影响阅读或边界的重构
- 新发现了失败模式、阻塞点或后续任务
- 改变了测试通过情况、评估结果或已知风险判断

CHANGELOG.md 应记录的不只是"做了什么"，还应记录：

- 何时完成
- 哪些方法奏效
- 哪些方法无效
- 哪些尝试失败、失败原因是什么
- 当前卡在什么地方
- 由本次工作引出的新任务
- 测试状态如何变化
- 是否发现新的风险或技术债

记录失败的方法尤其重要。对 Codex 来说，忘记失败尝试是重复劳动的主要来源。像"尝试使用 xxx，失败，原因是 xxx，后续应改为 xxx"这样的记录必须保留。

CHANGELOG.md 的写法应面向后续继续开发的人和模型：信息具体、可追溯、可用于下一轮决策，避免只写空泛完成项。

## 6. 编码风格：简单、清晰、可改、可排查

### 6.1 总体风格

默认优先写简单、直接、清晰的实现。代码首先应该易读、易改、易排查，而不是追求"架构感""面向未来的大设计"或某种教科书式模式。

不要把 Python 写成 Java。能用函数、模块、dataclass 和清晰组合关系解决的问题，不要强行引入复杂类层级、抽象基类体系或过重的设计模式。

抽象必须服务于当前真实复杂度，不要为了未来假设提前设计。没有两到三个以上真实使用点，不要急着做通用框架；没有明确边界痛点，不要急着引入 service/repository/framework 分层。

### 6.2 命名

命名必须准确表达语义。避免模糊和懒惰命名，例如 data、obj、manager、helper、util、misc、process_data 之类没有边界信息的名称。函数名应使用动词短语，类名应使用名词。布尔值命名应能直接读成判断句，例如 is_active、has_pending_tasks、should_retry。

如果一个名字不能帮助读者理解"这个东西负责什么、输入输出是什么、边界在哪里"，就应重命名。

### 6.3 函数与类

函数应尽量保持单一职责，不要在一个函数中同时混合决策、执行、状态修改、日志格式化和错误恢复逻辑。类必须边界清楚，避免形成集装配、状态管理、业务编排、对外 API 于一体的上帝类。

优先组合，不优先继承。一个类如果同时负责装配、状态、调度、执行和对外接口，通常说明它应该拆分。主入口类可以作为 assembly root 存在，但只应负责组装与协调，不应成为无限膨胀的总控中心。

### 6.4 数据结构

对于核心状态、跨模块传递对象、长期存在的数据，优先使用清晰显式的数据结构，推荐 dataclass。对于局部、短生命周期且语义简单的数据，可以使用 dict，但字段含义必须明确。

不要让关键领域对象长期以混乱裸 dict 的形式在系统中漂流。重要对象应具备明确结构、边界和版本意识，尤其是会进入状态持久化、恢复、工具接口或模型上下文的对象。

### 6.5 模块组织

模块按职责拆分。小项目先保持扁平；只有在复杂度真实上升后，再逐步分层。不要一开始就铺很重的目录结构。避免没有边界的 utils.py、helpers.py、misc.py。公共代码应优先按领域和职责归类，而不是扔进一个杂项模块。

模块依赖必须清晰，避免循环依赖、跨层乱 import 和隐式共享状态。

### 6.6 可读性与控制流

显式优于隐式。依赖、输入输出、副作用、状态变化都应尽量写清楚。少写难追踪的动态行为、魔法分发和元编程。控制流尽量线性，尽量通过小函数拆分复杂逻辑，而不是层层嵌套。

注释应解释"为什么这样做""约束是什么""这里依赖什么不变量"，不要只是翻译代码字面意思。

### 6.7 Pythonic 约束

优先使用：

- dataclass
- 小型纯函数
- 组合对象
- 显式映射分发代替冗长 if/elif
- 标准库

不要主动引入自己并不熟悉、且没有明确收益的重型框架或模式。不要为了所谓扩展性而平白引入工厂模式、复杂继承树、过多抽象基类、过度分层的服务结构。

### 6.8 针对 Agent 项目的额外约束

模型调用、工具调用、状态管理、prompt 构造和业务逻辑应尽量分开，不要混成一团。不要把模型输出直接当作系统事实；模型输出应先被结构化解析、验证，再进入系统状态。系统关键决策应尽量依赖显式状态，而不是散落在 prompt 历史中的隐式信息。

复杂配置不要全部平铺在一个超大配置类里，应按子域分组。新增代码时，优先与当前局部风格保持一致；除非明确要求，不要借机大规模重构周边代码。

默认输出风格是：先给出最小但完整的可用实现；除非复杂度已经明显上升，否则不要额外引入工厂模式、复杂继承体系、抽象基类海洋或多层 service/repository/framework 结构。

## 7. 禁止过度防御性编程，但系统必须强恢复

本项目默认不鼓励在普通业务逻辑中泛滥使用 try/except。错误应尽量自然暴露，而不是被静默吞掉。禁止空 except，禁止宽泛兜底后只打印日志，禁止吞错继续运行。

但"不鼓励防御性编程"不等于"不做输入校验、不做不变量检查、不做状态合法性约束"。减少的是无信息增益的异常吞噬和模糊兜底，不是减少明确失败和显式约束。

### 7.1 异常处理边界

集中式异常处理只应出现在以下边界层：

- 工具调用层
- 任务调度层
- 状态持久化层
- 外部 IO 边界

在内部核心逻辑中，除非满足以下条件，否则不要随意包裹 try/except：

- 需要补充上下文信息后重新抛出
- 需要将底层异常转换成更高层、结构化的系统错误
- 需要执行明确资源清理或补偿动作

### 7.2 错误必须可定位

每个错误都必须携带足够上下文，至少包括：

- 当前 step / action id
- 当前工具名（如适用）
- 输入参数摘要
- 当前状态版本号
- 最近 checkpoint id
- 调用栈
- 原始异常类型与消息

错误日志必须能直接回答：错在什么步骤、什么状态、什么输入下，而不是只留下"调用失败""发生异常"这类无用描述。
---

## 8. 可观测性：每一步都应可追踪

Deep Research Agent 的核心难题之一不是"能不能跑"，而是"出了问题能不能看清为什么错、错在哪、怎么复现"。因此可观测性是基础设施，不是锦上添花。

系统应逐步具备以下最小可观测能力：

- 每个 step 有唯一 id
- 每次模型调用有请求摘要、响应摘要、耗时、结果类型
- 每次工具调用有输入摘要、输出摘要、耗时、错误类型
- 每次状态变化有版本号，必要时有 diff 或变更摘要
- 每次 checkpoint 有唯一标识和创建时机
- 每次恢复都能追溯来源 checkpoint 和恢复理由

日志必须可按 task/session/step 检索。默认不要在 stdout 大量倾倒原始响应、完整上下文或巨型对象；需要详细诊断时，应进入专门日志文件或结构化 trace。输出给开发者和 Codex 的信息，优先是简洁摘要、关键指标和定位线索。

---

## 9. 不可过分依赖假工具

Agent 开发不能只依赖 fake / replay。
对于外部工具、检索系统、文件系统、网络 API、搜索质量和真实数据分布相关的问题，必须尽早通过真实调用验证。许多关键失败模式只有在真实环境中才会暴露，包括 schema 漂移、权限问题、限流、超时、分页、脏数据、低质量返回和语义无效返回。
fake / replay 主要用于加快局部迭代、稳定复现问题、验证内部逻辑和构建回归测试；真实调用则用于发现环境真实性问题、工具契约问题和任务级有效性问题。二者必须结合使用，不能互相替代。

---

## 10. 明确禁止的行为

以下行为默认禁止，除非有非常明确、被测试和文档支持的工程理由：

- 一次性引入完整大框架，而不是先做最小可验证实现
- 在没有测试的情况下修改 prompt、tool schema 或 state schema
- 为了通过测试而弱化测试本身，例如放松断言、删除失败样例、随意 skip
- 在业务逻辑中到处堆宽泛 try/except
- 修改工具接口但不给出兼容性测试或迁移说明
- 把恢复逻辑散落在业务分支中，而不是保持在运行时边界层
- 顺手进行与当前任务无关的大规模重构
- 把临时调试、运行过程文件、缓存、下载内容提交到仓库
- 让上帝类继续膨胀而不拆分边界
- 用隐式共享状态代替明确状态传递

---

## 11. 面向 Codex 的执行约束

Codex 在本项目中的默认行为必须偏保守、偏最小化、偏验证驱动。收到任务后，应先理解现有代码、测试和 CHANGELOG，再从最小失败测试入手，而不是直接铺写大量实现。默认流程如下：

先阅读相关模块、已有测试和最近的 CHANGELOG 记录；再写一个最小失败测试，或补一个能暴露当前问题的回归测试；然后用最小实现让该测试通过；接着运行 `pytest tests/ -q --fast` 检查没有回归；最后更新 CHANGELOG.md，记录这次修改、测试结果、有效方法、失败方法和新发现的任务。

若发现需要更大范围重构，应先判断是否真的由当前失败模式驱动。若不是，则不展开。若确有必要，应先把重构和功能变化分开，并保证每一步仍然小规模、可测、可回滚。

Codex 不应因为看到"未来可能需要恢复、扩展、多工具、多阶段编排"而主动先建完整平台。所有运行时能力都必须由当前真实问题驱动，以最小形式逐步进入系统。

---

## 12. 默认目录与测试组织要求

测试必须放在专门测试目录中，不允许与主代码混杂堆放。测试目录内部也要保持清晰分类，例如按模块、子系统或测试层级组织，而不是所有测试文件平铺且无边界。

推荐思路是：单元测试靠近模块语义划分，流程测试靠近最小 harness 划分，回归测试按失败模式或 bug 来源归档。重点不是目录形式是否绝对统一，而是让后续读者和 Codex 能迅速理解：某类测试在验证什么、覆盖哪一层边界、失败时应先看哪里。

---

## 14. 提交前检查清单

提交前必须至少确认以下事项：

- [ ] 当前改动只解决一个明确工作单元
- [ ] 所有新增模块先有测试文件
- [ ] 新 bug 先有复现测试再修复
- [ ] `pytest tests/ -q --fast` 已通过
- [ ] 如有必要，相关完整测试已通过
- [ ] CHANGELOG.md 已更新
- [ ] 没有把密钥、环境、缓存、运行过程文件带入提交
- [ ] 没有顺手引入不必要的大抽象或大重构
- [ ] 如改动了 prompt/tool/state 契约，已有相应契约测试
- [ ] 如改动影响行为，已记录验证结果而不是仅凭感觉判断

---

## 15. 最终要求

本项目追求的不是一次性设计出"完美 Agent 系统"，而是在持续迭代中建立一个简单、透明、可测试、可恢复、可定位、可继续演进的 Agent 系统。

所有开发都应遵循同一个核心方向：

**先最小化，后泛化；先验证，后扩展；先保留现场，后谈自动恢复；先用测试证明，再用抽象复用。**

如果某个设计让系统更难理解、更难测试、更难定位错误、更难从现场继续，那么它即使"更有架构感"，也不符合本规范。
