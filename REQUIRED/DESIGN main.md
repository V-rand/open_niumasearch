# Deep Research Agent 上下文管理与工作流设计说明（V0.1）

## 1. 文档目的

本文档用于定义 Deep Research Agent 的第一版需求与设计边界，目标是让工程同事可以据此开始实现，而不是继续停留在抽象讨论层面。

这个系统不是一个通用聊天机器人，也不是一个把所有历史都塞进上下文的大模型包装器。它的目标很明确：在长链路的深度研究任务中，把检索、阅读、整理、归纳、阶段总结与最终写作之间的关系管理清楚，让模型在长流程中尽量不乱、不丢、不早收工，并且让中间过程可追踪、可回溯、可交接。

本文档聚焦于以下问题：

* 研究阶段和写作阶段如何划分。
* 上下文该保留什么、卸载什么、外部化成什么文件。
* 搜索结果、网页正文、研究笔记、证据、TODO、阶段总结之间的边界如何定义。
* 如何防止模型过早关闭任务、过早给 TODO 打勾。
* 第一版系统应该做到什么，不应该一开始做什么。

本文档不是数据库设计说明，也不是 UI 文档，也不是 prompt 细节草案。第一版默认使用文件化的外部工作记忆，不依赖数据库、RAG 或向量检索。

---

## 2. 总体设计原则

### 2.1 核心思想：把上下文管理转化为“外部工作记忆管理”

系统的核心不是压缩聊天历史，而是把研究过程中真正重要的对象外部化管理。模型上下文只保留当前决策所需的活跃工作集，其余内容一旦已经沉淀为稳定文件，应尽量退出主上下文。

这意味着，系统管理的重点不是单纯的对话轮次，而是研究对象：

* 当前要解决的问题
* 当前 TODO 的局部切片
* 当前正在使用的来源
* 当前临时判断
* 当前研究笔记
* 当前阶段结论和未解决项

因此，第一版系统不采用“统一长上下文 + 压缩总结”的思路，而采用“文件化分层对象 + 阶段性 checkpoint”的思路。

### 2.2 第一版采用顺序主流程，而不是研究写作完全并行

V1 采用以下主流程：

1. 先走完一次完整的研究阶段。
2. 当研究达到“足以支撑初稿”的程度后，切换到写作阶段。
3. 写作阶段允许主动补充检索，但这种检索是局部补证，不重新打开一个无限扩张的大研究流程。

换句话说，V1 是一个“研究优先、写作为后、写中可补证”的系统。

我们不在第一版中实现研究与写作的双主流程并行，也不引入双平级 agent 协同。因为第一版更重要的是把对象边界和状态流转做扎实，而不是把系统做复杂。

### 2.3 研究与写作必须分账，不允许共用同一套 TODO 或同一套工作文件

研究阶段和写作阶段不是同一件事。研究阶段处理的是问题空间、来源空间、证据采集和判断形成；写作阶段处理的是章节组织、论证结构、正文推进和证据装配。

因此，系统必须做到：

* 研究 TODO 和写作 TODO 分开管理。
* 检索索引和原文归档属于 research 域，不属于 writing 域。
* 写作阶段消费的是研究阶段沉淀下来的研究笔记与证据，而不是直接消费搜索摘要或原始网页正文。
* 写作阶段如果需要补证，可以发起局部补检索，但不能覆盖研究主轨迹。

## 3. 范围与非目标

## 3.1 第一版范围

第一版必须实现以下能力：

1. 支持研究阶段的分阶段推进和 subgoal 管理。
2. 区分搜索命中、已读网页正文、研究笔记、稳定证据、阶段总结这几类对象。
3. 支持 TODO 的显式管理，并且防止模型过早给任务打勾。
4. 支持在研究阶段持续生成中间产物，而不是等到最后一次性综合。
5. 支持研究阶段结束后切换到写作阶段。
6. 支持写作阶段局部补证，但补证以追加记录的形式进入系统，而不是改写原研究主轨迹。
7. 支持用文件系统作为外部工作记忆进行管理。

## 3.2 第一版非目标

第一版明确不做以下内容：

* 不做数据库或向量库驱动的长期记忆系统。
* 不做复杂的 RAG 检索编排。
* 不做多平级 agent 自主协同。
* 不做自动事实判定器或外部审稿器。
* 不做复杂的多用户协作权限系统。
* 不做完整的 UI 产品设计。
* 不把 prompt 编排设计成主要交付内容。

## 4. 工作流总览

## 4.1 主阶段定义

系统包含两个主阶段：

### 阶段 A：Research Phase

目标是围绕目标问题完成信息收集、来源筛选、原文阅读、研究笔记沉淀、阶段证据整理，并达到“可以开始写作”的程度。

该阶段不是简单的搜资料，而是必须产生以下中间产物：

* 子问题摘要
* 局部发现
* 研究笔记
* 阶段性结论
* claim–evidence 对齐记录的初稿
* 可供后续写作消费的研究资产

### 阶段 B：Writing Phase

目标是基于研究阶段产物完成结构化写作。

写作阶段以章节推进和论证表达为主，不再无限扩展研究范围。但如果某段写作暴露出证据缺口，允许发起局部补证。补证完成后以追加材料的形式回流写作阶段，不覆盖原研究轨迹。

## 4.2 阶段切换原则

Research Phase 结束后，系统不直接把所有对话历史带入 Writing Phase，而是应输出一个“写作输入包”。

这个输入包至少应包含：

* 研究阶段总体结论摘要
* 已验证或相对稳定的研究笔记
* 可引用的证据集合
* 已知冲突与不确定性列表
* 不再继续追踪的失败路径摘要
* 研究阶段 TODO 的最终状态

Writing Phase 基于这个输入包开始，而不是基于研究阶段的全部动作历史开始。

## 5. Research Phase 的对象模型

研究阶段不是一个统一的信息池，而应拆成若干职责明确的对象层。

### 5.1 Source Index（来源索引）

来源索引用于管理“发现了什么来源”。它记录的是候选线索和来源状态，而不是正式证据本身。

典型实现可以是 `search.md`，但其职责必须被严格限制：

* 记录每个 URL 或来源条目的基本信息
* 记录其来源于哪个 query 或哪个工具调用
* 记录标题、站点、作者、发布时间等元数据
* 记录其当前状态：仅命中、已选择阅读、已读取正文、已进入研究笔记、已进入证据层、已放弃
* 如果已用 jina 读取正文，记录 raw 文件路径
* 如果该来源服务于某个特定 subgoal，记录所属 subgoal
* 如有必要，记录简要价值说明和分类标签

重要约束：

* Source Index 只做来源总索引，不做研究笔记，不做写作草稿，不做正式论证。
* 搜索 API 返回的 `url + summary` 只是候选线索，不等于证据。
* 直接回答题目的 benchmark 页面也只算高价值线索，不能直接视为最终证据。

### 5.2 Raw Source（原文归档）

Raw Source 用于保存通过 jina 等方式读取到的网页正文。

其职责是保存“原始、可回溯的材料”，而不是保存模型理解后的结论。通常放在 `raw/` 目录中。

Raw Source 的要求如下：

* 一篇来源一个独立文件，文件命名应可映射回 Source Index 条目
* 文件中应保留基础元数据头，例如来源 URL、抓取时间、原始标题、作者、发布时间等
* 原文应尽量保持原始内容，不混入过多解释性文字
* Raw Source 不承担阶段总结、证据归纳、写作草稿等职责

Raw Source 的意义在于：

* 让后续判断可以回溯到原材料
* 避免大量正文长期占据主上下文
* 为证据抽取和研究笔记提供上游材料

### 5.3 Research Notes（研究笔记）

Research Notes 是研究阶段最关键的中间层，用于记录“这些材料对当前问题意味着什么”。

这层对象必须显式存在，不能再让系统把解释性内容混在 Source Index 或阶段总结里。

Research Notes 的职责包括：

* 围绕当前问题组织阶段性理解
* 记录对多个来源的比对结果
* 记录模型目前形成但尚未最终封存的判断
* 标记哪些判断较稳定，哪些判断还需要补证
* 为后续写作准备结构化笔记

Research Notes 可以包含两类内容：

1. 问题笔记：围绕研究问题本身组织，例如时间线、定义差异、争议点、因果链等。
2. 写作预备笔记：围绕未来表达组织，例如某节可用素材、某个 claim 只能弱表述、某段可放在背景部分等。

Research Notes 仍然属于 research 域，不属于正式写作草稿。

### 5.4 Evidence Record（证据记录）

Evidence Record 用于记录已经足够稳定、可以为后续写作或结论提供支撑的证据。

来源被读过，不等于证据已经形成。研究笔记形成了，也不等于已经有稳定证据。系统必须把“已读材料”和“稳定证据”分开。

每条 Evidence Record 至少应包含：

* 一条要支撑的 claim 或判断
* 支撑该 claim 的来源位置（raw 文件路径或来源片段定位）
* 证据强度或可靠性备注
* 是否存在冲突来源
* 对该证据适用边界的说明

Evidence Record 不是写作正文，但它比研究笔记更稳定，是写作阶段可以直接消费的对象。

### 5.5 Subgoal Checkpoint（阶段沉淀）

每个 subgoal 完成时，系统必须产出一个 checkpoint，而不是简单清空上下文。

Checkpoint 至少包括以下内容：

* 该 subgoal 原始目标
* 本阶段采取的关键动作轨迹摘要
* 关键 query、关键来源类型、关键文件路径摘要
* 形成了哪些有效研究笔记
* 形成了哪些可用证据
* 当前结论是什么
* 哪些问题尚未解决
* 哪些路径尝试后被放弃，以及放弃原因
* 对后续 TODO 的调整建议

Checkpoint 的作用是把一个 subgoal 内部冗长、临时、混乱的过程沉淀成一个可以继续被消费的稳定对象。

## 6. Research Phase 的上下文管理规则

## 6.1 不按轮次管理上下文，按“活跃对象”管理上下文

上下文管理不能再简单理解成“保留最近几轮聊天”。对于 deep research，真正需要保留的是当前活跃工作集。

所谓活跃工作集，通常包括：

* 当前 subgoal 的问题定义
* 当前 research TODO 的相关局部
* 当前正在比较和引用的 2–4 个活跃来源
* 最近形成但尚未固化的临时判断
* 当前正在更新的研究笔记片段
* 少量最近工具调用摘要，用于保持动作连续性

除此之外，其他内容只要已经稳定写入外部文件，就应尽量退出主上下文，只保留路径、标签和摘要。

## 6.2 Subgoal 内允许相对宽松的上下文，但要及时外部化正文

在一个 subgoal 内，系统可以相对宽松地保留较完整的研究历史，不必一开始就强力压缩。

但对通过 jina 获取的大段网页正文，应尽快外部化到 `raw/` 文件中。主上下文中只保留：

* 来源标识
* 文件路径
* 简要摘要
* 当前正在使用的少量原文摘录或结论

第一版不要求做复杂的 token 最优调度，但必须做到“大段原文不长期驻留在主上下文中”。

## 6.3 Subgoal 内部也存在三个认知阶段

一个 research subgoal 内部，通常会经过三个不同状态：

### 探索态

问题还没完全收敛，query 偏发散，来源还在大范围筛选。

### 聚焦态

关键来源已经出现，系统进入阅读、比较、归纳和判断形成阶段。

### 沉淀态

开始把来源、临时判断和研究笔记整理成稳定的 checkpoint 和证据记录，以便后续继续推进。

第一版不要求在程序层面显式实现这三个状态机，但实现时应默认存在这种阶段感，并为沉淀态提供明确出口。

## 7. TODO 系统设计

TODO 不是一个简单的待办列表，而是研究阶段与写作阶段各自的显式控制面板。它的主要作用不是“列计划”，而是提供一个可跟踪、可调整、可验证的任务状态对象。

## 7.1 研究 TODO 与写作 TODO 必须分开

Research TODO 负责管理：

* 还要查哪些问题
* 哪些来源需要确认
* 哪些证据缺口需要补齐
* 哪些争议点需要澄清

Writing TODO 负责管理：

* 章节怎么推进
* 哪一节缺支撑
* 哪个 claim 需要弱化或补证
* 哪段需要改写、重写或调整结构

这两个列表不能混合，不能互相覆盖。

## 7.2 TODO 不应被程序机械顺序执行

程序不应该把 TODO 视为脚本，然后机械地一项一项跑。更合理的做法是：

* TODO 是显式外部状态对象
* 模型每轮读取 TODO，并结合当前研究状态决定做哪一步
* 模型执行完动作后，通过结构化操作更新 TODO
* 程序负责读写接口和状态约束，而不是代替模型做研究判断

因此，TODO 更像是模型的“研究控制面板”，而不是执行引擎。

## 7.3 TODO 建议做成独立 skill 或结构化操作接口

相比让模型直接编辑 markdown 文本，TODO 更适合作为一组显式的结构化操作来管理。

例如支持以下操作：

* create
* split
* reprioritize
* update_status
* attach_note
* attach_evidence
* defer
* abandon
* reopen
* close_attempt

这样可以避免模型在自由文本里随意改动，增强状态一致性。

## 7.4 防止模型过早打勾：禁止二元 done/not-done 语义

系统必须防止模型过早关闭任务。第一版不应采用简单的 `done / not done` 状态。

建议最少使用以下状态：

* `open`
* `in_progress`
* `tentatively_resolved`
* `closed`
* `deferred`
* `abandoned`

其中，`tentatively_resolved` 非常关键。它表示“目前看像是解决了，但尚未满足最终关闭条件”。

这可以显著降低模型因为“感觉差不多了”而过早 `closed` 的概率。

## 7.5 关闭任务必须走 closure attempt，而不是直接打勾

模型不应拥有直接把任务标为 `closed` 的自由。更合理的做法是：

* 模型只能发起 closure attempt
* closure attempt 必须附带关闭理由和外部对象引用
* 程序根据规则检查是否允许进入 `closed`
* 如果条件不足，则只能进入 `tentatively_resolved` 或保持 `in_progress`

### 关闭条件建议

一个任务要被真正关闭，至少应满足：

1. 结论已形成，且不是纯搜索行为结论。
2. 至少绑定了一条外部对象，例如研究笔记或证据记录。
3. 若存在冲突，冲突已被记录。
4. 若存在未决项，应进入 `tentatively_resolved`，而不是 `closed`。
5. 若该任务有子任务，子任务状态应满足收敛要求。

### 关闭说明建议

每次 closure attempt 建议要求模型输出一个短模板，例如：

* 结论：
* 依据：
* 未决项：

如果模型写不出这一小段，通常说明任务还不应该被关闭。

## 7.6 TODO 项应尽量写成目标型，而不是动作型

例如：

不推荐：

* 查一下某人背景
* 搜几篇文章
* 看看有没有来源

推荐：

* 确认某角色起源设定，并至少找到两条可交叉支持的来源
* 比较 A 与 B 的发布时间差异，并记录冲突解释
* 为章节 2 的 claim-3 补齐一级来源，否则将表述降级

动作型 TODO 容易导致模型一执行就打勾，目标型 TODO 更适合结构化关闭。

## 8. Writing Phase 设计

## 8.1 Writing Phase 的输入

Writing Phase 不直接继承 Research Phase 的全部动作历史，而是继承研究阶段沉淀出来的写作输入包。

Writing Phase 至少应接收以下输入：

* 研究总摘要
* 研究笔记
* 可用证据集合
* 已知冲突和风险点
* 研究阶段未解决但可接受的限制

## 8.2 Writing Phase 的主要对象

写作阶段至少应维护以下对象：

* 章节草稿
* 局部段落草稿
* 写作 TODO
* claim–evidence 对齐记录
* 补证请求记录

这些对象属于 writing 域，不应与 research 域文件混写。

## 8.3 Writing Phase 可主动补证，但补证必须受控

写作过程中，如果发现某段证据不足，可以发起局部补证。这个补证不是重新打开整个 Research Phase，而是一个受控的小回流。

补证应遵循以下规则：

* 补证请求在 writing 域产生
* 补证动作可调用 research 能力
* 补证结果作为追加材料进入系统
* 不覆盖原有研究主轨迹，只在研究资产上追加新记录

这样可以保持主研究过程的稳定性，同时允许写作阶段提升质量。

## 9. 研究过程中持续写“中间稿”的原则

系统不会等所有检索都结束之后才开始形成文本。研究过程中应持续生成中间产物。

这些中间产物包括但不限于：

* 子问题摘要
* 局部发现
* 研究笔记
* 章节素材草案
* claim–evidence 初步对齐记录

这样做的原因有两个：

第一，可以降低最后一次性综合的负担。研究完成后，系统已经有大量半成品，而不是从零开始写。

第二，可以尽早暴露证据不足的问题。很多时候，只有开始尝试写一段说明时，才会发现某个判断缺少强支撑。

注意：这些“中间稿”发生在 research 阶段，但它们依然属于 research 域中的写作准备资产，不等于正式写作稿。

## 10. 文件组织建议（V1）

以下是第一版建议的文件结构，仅作为实现参考，具体命名可调整，但对象边界不应被打破。

```text
project/
  research/
    search.md
    todo.json
    raw/
      <source_id>.md
    notes/
      <subgoal_id>.md
      <topic>.md
    evidence/
      <evidence_id>.md
    checkpoints/
      <subgoal_id>.md
    summaries/
      research_overview.md
  writing/
    todo.json
    outline.md
    drafts/
      section_01.md
      section_02.md
    claim_evidence/
      ledger.md
    backfill_requests/
      <request_id>.md
```

这个结构表达的不是目录美观，而是职责分离：

* `research/search.md`：来源索引
* `research/raw/`：原始正文归档
* `research/notes/`：研究笔记
* `research/evidence/`：稳定证据
* `research/checkpoints/`：subgoal 阶段沉淀
* `research/todo.json`：研究 TODO
* `writing/todo.json`：写作 TODO
* `writing/drafts/`：正式写作稿
* `writing/claim_evidence/`：论证对齐记录
* `writing/backfill_requests/`：写中补证请求

## 11. 与上下文整理器（Context Curator）的关系

第一版暂不引入第二个平级 agent。系统只采用单主 agent 驱动任务。

后续如果发现上下文整理确实需要单独抽离，可以考虑引入一个弱权限的 Context Curator，但其职责只能是：

* 检查对象状态是否清晰
* 建议哪些内容应保留在上下文中
* 生成 checkpoint 草案
* 提示证据缺口、冲突和冗余

它不应拥有以下权限：

* 不应修改正式结论
* 不应新增或关闭 TODO
* 不应主动发起检索
* 不应直接改写写作草稿

该角色不在 V1 范围内，仅作为后续可选增强方案。

## 12. 典型执行流程（V1）

以下给出一个典型执行流程，帮助工程实现理解对象流转关系。

### 12.1 研究启动

系统接到一个 deep research 任务，创建 research 目录，初始化 research TODO，并生成第一个 subgoal。

### 12.2 来源检索

模型围绕当前 subgoal 发起 query，检索结果以 `url + summary` 形式进入 Source Index。此时这些结果只是候选来源。

### 12.3 选择性阅读

模型评估哪些来源值得进一步阅读。只有被选中的条目才使用 jina 读取正文，并将正文存入 `raw/`。

### 12.4 形成研究笔记

模型基于 raw 内容和多个来源比对，形成 Research Notes。此时上下文中只保留当前活跃来源与少量必要摘录，其他内容已转入文件。

### 12.5 形成证据与阶段结论

当某些判断足够稳定时，创建 Evidence Record，并更新研究笔记。若 subgoal 已基本完成，则生成 Checkpoint。

### 12.6 更新研究 TODO

模型基于 checkpoint 对研究 TODO 做增删改：

* 关闭满足条件的任务
* 将不完整但接近完成的任务设为 `tentatively_resolved`
* 对需要继续追踪的问题拆分新任务
* 对价值不高的方向设为 `deferred` 或 `abandoned`

### 12.7 进入写作阶段

当研究达到写作阈值后，生成 research overview 和写作输入包，初始化 writing 目录与 writing TODO，开始正式写作。

### 12.8 写作阶段局部补证

若写作暴露证据缺口，则生成 backfill request，调用研究能力补充证据，并将新增材料追加回 writing 阶段消费对象中。

## 13. 第一版实现优先级建议

为了降低工程复杂度，建议按以下顺序实现：

### 第一优先级

* 文件对象边界
* Research TODO / Writing TODO 分离
* Source Index / Raw / Notes / Checkpoint 的最小工作流
* TODO 的多状态管理与 closure attempt 机制

### 第二优先级

* Evidence Record 的结构化沉淀
* 写中补证请求机制
* Claim–evidence 对齐记录

### 第三优先级

* 自动化的上下文打包建议
* Context Curator
* 更细的可靠性评分或冲突检测

这意味着第一版应优先把“对象和状态管理”做好，而不是追求更智能的自动化。

## 14. 当前明确的关键结论

截至目前，本项目的关键共识如下：

1. 这个系统的重点不是压缩聊天历史，而是管理文件化的外部工作记忆。
2. 第一版采用顺序主流程：先研究，再写作，写作中允许局部补证。
3. 研究和写作必须分账，不能共用 TODO，不能混用文件。
4. 搜索命中、网页正文、研究笔记、稳定证据、阶段 checkpoint 必须分层管理。
5. 研究阶段不是只搜资料，而是要持续产出面向写作的研究笔记和中间材料。
6. 上下文管理按活跃对象进行，而不是按最近轮次进行。
7. TODO 不应由程序机械执行，而应由模型自主推进，但必须通过显式状态机和规则约束来防止过早关闭。
8. 第一版暂不引入第二个平级 agent。

## 16. 最终定义

可以用一句话概括本系统：

**这是一个以研究为主、写作为后、写中可补证的 Deep Research 工作流系统。它通过文件化的多层研究资产来管理长链路任务中的上下文，把模型的主上下文限制在当前活跃工作集上，并通过显式 TODO 状态机与阶段 checkpoint 来维持过程可控、可追踪、可回溯。**
