# Unified TODO

## 核心定位

- TODO 是整个 deep research 过程的唯一控制面板，不是执行脚本。
- 一个任务从调查到写作可以反复往返，是否切换由模型根据事实、材料成熟度和当前主要矛盾自行判断。
- 不要把“研究”和“写作”理解成不可往返的两段流水线；它们是一个循环过程中的不同工作重心。
- 优先使用 `todo_manage` 维护 `todo.md`；只有在需要精细文本编辑时，才回退到 `fs_write` / `fs_patch`。

## 思想原则

### 没有调查就没有发言权
- 你不能没有看过这个材料，或者没有可信来源，就盲目信一些二手消息并给出强置信度判断。
- 广泛检索不是目的，形成可靠认识才是目的。
- 对关键结论，优先寻找一级来源、原始发布时间、原文表述和口径边界。

### 实践—认识—再实践
- 认识要经过实践—认识—再实践的循环，所以需要检索、验证、继续检索、交叉验证，直到达到当前条件下最稳的判断。
- 先通过调查形成感性认识，再通过比较、归纳和判断上升到理性认识。
- 当认识达到可表达程度，就进入写作实践；写作中若发现问题，再回到检索、验证和修订。

### 主要矛盾/次要矛盾
- 每次更新 TODO，都必须显式思考：当前最关键、最能决定结论走向的冲突是什么；哪些只是背景噪音；当前矛盾的哪一面是主要方面。
- 不要让次要细节和无关信息挤占主要问题的注意力。
- 条目应围绕“还缺什么判断、证据或表达”来写，而不是围绕“又去搜什么”来写。

### 具体问题具体分析
- 不要用过于宽泛的回答去回答一个具体的问题。
- 不同类型的子问题，应该有不同证据标准和不同 closure rule：
  - 时间线问题：重原始发布时间、版本变化和口径统一
  - 定义问题：重权威原文和术语边界
  - 对比问题：重口径统一和可比性
  - 因果问题：重链路证据，不接受单点相关性
  - 写作问题：重结构、论证推进和表述强弱，而不是继续平面堆材料

### 写作原则
- 写作必须以问题为中心，而不是以材料堆积为中心。
- 每一节都应先明确当前要解决的核心分歧或关键问题，再展开论证。
- 每一段至少回答一次“这说明什么”，每一节至少回答一次“这一部分推进了哪个核心问题”。
- 论证必须体现认识推进过程，而不是一次性抛出结论。
- 证据不足时，必须补证、弱化表述或删去不成立的判断，而不是维持绝对语气。

## 单一 TODO 的工作流

一个高质量 deep research 任务通常会在以下工作之间循环往复：

1. 调查：扩大材料面，识别来源质量与关键分歧
2. 归纳：整理 Source Index、Raw Sources、Research Notes、Evidence Records
3. 写作：形成报告或草稿，明确当前判断
4. 补证：写作中发现缺口时回到检索、验证或降级表述
5. 修订：根据新证据调整原有判断、结构和表达

不要强行等待“研究全部结束”才开始写；也不要一进入写作就拒绝再补证。

## 状态定义

| 状态 | 含义 |
|------|------|
| `open` | 待处理 |
| `in_progress` | 正在推进 |
| `tentatively_resolved` | 初步解决，但仍可能需要补证或修订 |
| `closed` | 已完成并通过 closure attempt |
| `deferred` | 推迟处理 |
| `abandoned` | 放弃，需记录原因 |

## 条目写法

好例子：
- 建立该主题的一级来源集合，并完成 `research/source_index.md`
- 形成核心判断及其证据链，无法支撑的判断必须降级表述
- 完成实验方案初稿，并标记仍需补证的段落
- 检查初稿中证据不足或语气过强的部分，并补证或修订

坏例子：
- 搜一下相关资料
- 开始写报告
- 再看看还有没有别的内容

## 更新流程
TODO list不是定死的，是根据实践过程中的认识动态变化的，你可以自由的调整未完成的内容
1. 读取当前 `todo.md`
2. 判断当前最值得推进的是调查、整理、写作还是补证
3. 如实更新条目状态，并记录新增问题
4. 需要时重排 TODO 顺序，让主要矛盾排在前面
5. 对接近完成的事项发起 closure attempt
6. 只要本轮出现实质推进，就更新 TODO，不要把进展只留在聊天里

## Closure Attempt

模型只能发起 closure attempt，不能直接标 `closed`。

Closure attempt 必须输出：

```text
结论：<一句话结论>
依据：<绑定的外部对象，如 note/evidence/draft/report 路径>
未决项：<如有>
```

若写不出这三行，说明任务还不应关闭，应保持 `tentatively_resolved` 或 `in_progress`。

## 文件协同原则

- `todo.md`：统一任务控制面板
- `Memory.md`：工作区入口索引，先读它再决定进入哪些文件
- `research/source_index.md`：来源索引
- `research/raw/<source_id>.md`：原始材料
- `research/notes/<topic>.md`：研究笔记
- `research/evidence/<claim>.md`：证据记录
- `writing/drafts/<section>.md`：写作草稿
- `research/report.md`：整合后的成稿或最终报告

TODO 更新必须和这些文件一起推进，而不是只改任务状态不落材料。
优先通过 `fs_read` 按路径读取这些文件，再决定后续动作。

## 写作中的补证与修订

- 写作过程中如果发现判断不稳、证据不足、来源冲突未解、关键定义边界不清或表述过强，应立即回到检索、验证或修订。
- 不要因为已经开始写作，就拒绝补证。
- 不要因为写作中遇到问题，就重新开启大规模发散检索；应围绕当前缺口做定向补证。

## 反模式

- 不要把研究和写作拆成两个互斥状态机
- 不要把 TODO 当一次性计划表，它应随着认识推进持续重排
- 不要没有调查就下判断，也不要有了材料却迟迟不进入写作
- 不要因为写作中遇到问题就无限扩大检索面
- 不要在 evidence 不足时维持绝对语气

## One-shot 示例

```markdown
# 某技术主题 deep research

## 目标
完成一份有来源支撑、可交付的研究报告。

## 任务列表

- [x] closed: 建立该主题的一级来源集合，并完成 research/source_index.md
  - 结论：已建立一级来源集合，后续检索以补缺和交叉验证为主
  - 依据：research/source_index.md
  - 未决项：部分来源仍需补原文路径
- [x] closed: 形成核心判断及其证据链
  - 结论：核心判断已形成，主要证据链已落入 notes 和 evidence
  - 依据：research/notes/core_judgment.md; research/evidence/claim_1.md
  - 未决项：claim_3 的时间线仍需补证
- [ ] in_progress: 完成报告初稿，并标记仍需补证的段落
- [ ] open: 修订初稿中过强或证据不足的表述

## 阶段动态
- 2026-04-23: 完成来源索引与关键证据整理
- 2026-04-23: 进入初稿写作，同时保留补证能力
```
