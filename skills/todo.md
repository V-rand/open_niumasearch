<skill_identity>
你将维护一个 `todo.md`，但它不是线性项目管理表，也不是每一步都要修改的官僚文件。

它是研究控制面板：帮助你在“实践、认识、再实践”的循环中记录当前信念、最大不确定性、下一组行动和周期性更新。
</skill_identity>

<core_rules>
1. **周期维护，而非动作维护**：不要每个工具调用后都改 TODO。每个工作周期开始时计划，结束时更新。
2. **信念账本**：记录当前最可信判断、置信度、关键证据、反证或不确定点。
3. **信息增益原则**：下一组行动必须说明为什么值得做；如果说不清，就应写 synthesis、修订报告或停止。
   搜索和阅读必须服务于目标，不能为了显得勤奋而继续扩散。
4. **允许螺旋式回环**：写作中发现证据缺口可以回到检索；检索中形成阶段性认识可以直接写 notes 或 draft。
5. **Markdown 优先**：继续使用 `todo.md`、`leads.md`、`source_index.md`、`notes/*.md`，不要把状态管理外包给 JSON 或数据库。
6. **证据锚点**：source_index 中关键证据必须包含 `>` 块引用、URL/raw_path、以及该证据支持或反驳的判断。
7. **线索状态流转**：`leads.md` 中的线索应在周期更新中从 `[Unread]` 流转到 `[Read]`、`[Rejected]` 或 `[Used]`。
8. **交付形态适配任务**：短答案 benchmark 不需要长文；长答案、阅读清单、实验方案才需要更完整结构。TODO 中应写清本任务的可验证产出形态。
9. **认证后再闭合**：只有在产出已经落地后才能闭合；多 Agent 模式下，研究产出还需要 evaluator 认证后才交给 Writer。
</core_rules>

<template>
建议使用轻量结构：

```markdown
# FIREWALL
- 不可违反的用户约束
- 不可使用或必须谨慎使用的来源/方法

# 当前信念
- 主判断：...
- 置信度：低/中/高
- 支撑证据：...
- 主要反证或不确定性：...

# 最大不确定性
- 当前最可能改变结论的问题是什么？

# 本周期行动计划
- 当前小目标：...
- [ ] 行动 1：...
  - 预期信息增益：...
- [ ] 行动 2：...
  - 预期信息增益：...

# 周期更新
- 新证据：...
- 信念变化：...
- leads 状态变化：...
- 下一步：continue / revise / stop，理由：...
```
</template>

<source_index_contract>
精读后的来源不要只写摘要。至少保留能承重的证据块：

```markdown
## S1. Title
- url: ...
- raw_path: research/raw/...
- status: used

> 原文或接近原文的关键摘录。

- 含义：...
- 支持/反驳：...
- 后续用途：...
```
</source_index_contract>

<anti_patterns>
1. **只写不读的 leads**：搜索结果堆在 `leads.md`，但后续不标记、不筛选、不回看。
2. **模糊 source_index**：只有高度概括摘要，没有 `>` 原文证据块。
3. **单一巨型 note**：所有机制、变量、冲突都塞进一个文件，导致后续复读成本高。
4. **Cosplay 引用**：报告里有上标或作者年份，但 References 无法映射回 source_index/raw。
5. **无限检索**：已经无法说明下一步的信息增益，却继续调用搜索或阅读工具。
</anti_patterns>
